Q: Customer segmentation or scrubbing ?

Yes. **Customer segmentation and customer scrubbing are closely related but different activities.** Based on your RCM experience, you've likely worked with both.

### 1. Customer Segmentation

Segmentation means **dividing customers into meaningful groups** for targeted campaigns.

Examples:

* Premium customers
* New customers
* High-value investors
* Inactive customers
* Gold loan customers
* Customers in Hyderabad

Example SQL:

```sql
SELECT * FROM customers
WHERE city='Hyderabad'
AND investment > 500000;
```

A campaign is then sent only to that segment.

---

### 2. Customer Scrubbing

Scrubbing means **cleaning and validating the customer list before launching a campaign**.

Typical scrubbing includes:

* Removing duplicate mobile numbers
* Removing invalid phone numbers or email IDs
* Removing customers who have opted out (DND/consent)
* Removing blocked or inactive users
* Excluding blacklisted customers
* Removing recently contacted customers (frequency capping)

Example:

```text
Input List: 1,000,000 customers

↓ Remove duplicates
950,000

↓ Remove invalid numbers
920,000

↓ Remove DND customers
850,000

↓ Remove unsubscribed users
830,000

Final campaign audience = 830,000
```

Q:One database with shards ie userid,but user pumped 500m messages on the same time ?
If your sharding key is **userId**, and **one user suddenly sends 500 million messages at the same time**, you've hit the classic **hot shard (hot partition)** problem.

### Problem

```
Shard = hash(userId)

User A ─────────► Shard 12 (500M messages)

User B ─► Shard 3
User C ─► Shard 5
User D ─► Shard 8
```

Shard 12 becomes overwhelmed:

* CPU reaches 100%
* Disk I/O spikes
* Connection pool exhausts
* Queue backlog grows
* Other shards are mostly idle

This is known as **data skew** or a **hot partition**.

---

## Solutions

### 1. Composite Sharding Key (Most Common)

Instead of:

```
Shard = hash(userId)
```

Use:

```
Shard = hash(userId + campaignId)
```

or

```
hash(userId + bucket)
```

Example:

```
User A

Campaign1 → Shard 2
Campaign2 → Shard 8
Campaign3 → Shard 15
```

The same user's workload is spread across multiple shards.

---

### 2. Virtual Shards

Instead of 16 physical shards:

```
4096 virtual shards

Virtual Shard 101 → Physical 1
Virtual Shard 102 → Physical 7
Virtual Shard 103 → Physical 12
```

Large users occupy many virtual shards, making rebalancing easier.

---

### 3. Bucketing

Instead of:

```
userId
```

Use:

```
userId + bucket
```

Example:

```
500M messages

Bucket 0
Bucket 1
Bucket 2
...
Bucket 99
```

Now each bucket holds about **5 million** messages, distributing load.

---

### 4. Queue First, Database Later

```
Producer
     │
Kafka
     │
100 Consumers
     │
Database
```

Kafka absorbs the traffic burst, and consumers write to the database at a controlled rate, preventing overload.

---

### 5. Rate Limiting

If one customer exceeds an agreed throughput:

* 10K messages/sec → allowed
* 1M messages/sec → throttle
* Queue the excess traffic

This protects the platform from a single tenant monopolizing resources.

---

### 6. Auto-Splitting Hot Shards

Detect a hot shard:

```
Shard 12

↓

Split

Shard 12A
Shard 12B
Shard 12C
```

Systems like **Google Spanner**, **CockroachDB**, and **DynamoDB** automatically split heavily loaded partitions.

---

### 7. Time-Based Partitioning

Partition by both user and time:

```
userId + YYYYMMDD

UserA_20260714
UserA_20260715
```

This keeps historical data separated and reduces contention for long-lived users.

---

## How large messaging platforms handle this

Platforms such as WhatsApp, Uber, and LinkedIn generally avoid sharding solely by `userId`. They combine several strategies:

* Consistent hashing with virtual nodes
* Composite partition keys
* Kafka or Pulsar for buffering
* Dynamic partition rebalancing
* Auto-splitting hot partitions
* Per-tenant rate limiting
* Separate read and write paths

This ensures that even if a single customer sends **500 million messages**, the workload is distributed across many partitions instead of overwhelming one database shard.

**Interview takeaway:** If asked, "What if one user generates 500 million messages but you're sharding by `userId`?", the key answer is: **`userId` alone is a poor sharding key for write-heavy systems because it creates hot partitions. Use composite keys, bucketing, virtual shards, and an event queue (Kafka/Pulsar) to distribute the load and protect the database.**

This is an important distributed systems question. The answer depends on your access pattern.

If you **write using `(userId + bucket + date)`** but later need to **read using only `(userId + date)`**, you need an additional strategy.

## Option 1: Query All Buckets (Most Common)

Suppose you created 100 buckets.

Write:

```text
(userId, date, bucket)
```

```

 ├── Bucket 0
 ├── Bucket 1
 ├── Bucket 2

```

To read:

```sql
SELECT *
FROM messages
WHERE userId = ?
AND date = ?
```

The application fans out the query to all 100 buckets in parallel and merges the results.

This is exactly how many distributed databases work.

---

## Option 2: Secondary Index

Primary table:

```text

(userId + bucket)
```

Secondary index:

```text
(userId, date)
→ bucket list
```

Read flow:

```

      │

      │
Buckets = [3,8,11]
      │
Read only those buckets
```

Instead of scanning all 100 buckets, you only query the buckets that contain data.

---

## Option 3: Kafka Offset / Metadata Table

Maintain metadata:

```

2026-07-14

Buckets:
1
7
18
52
```

The read service first fetches the metadata, then queries only those partitions.

---

## Option 4: Time-Based Partitions

If users mostly read today's data:

```
userId
+
date
+
bucket
```

A request for a single day only scans that day's buckets, not the entire history.

---

## Which approach is best?

For a messaging platform handling hundreds of millions of messages:

* **Writes:** `userId + date + bucket` to distribute load.
* **Reads:** Use a **metadata/secondary index** to identify which buckets to query.
* Execute those bucket reads **in parallel** and merge the results.

This avoids both a hot shard on writes and an expensive full scan on reads.

### Interview follow-up

If an interviewer asks, "Won't querying multiple buckets make reads slower?", a strong answer is:

> "Yes, there's a fan-out cost, but we trade a small increase in read complexity for massive write scalability. We minimize the fan-out by maintaining a metadata index of active buckets and issuing parallel asynchronous reads, which keeps latency low while preventing hot partitions."

Q:Within the shard,do we need to create a buckets?
Yes—but **not always**. It depends on what problem you're solving.

If your shard key is `userId` and a single user generates **500 million messages**, then **creating buckets only inside the same shard does not solve the hot shard problem**.

### Case 1: Buckets within the same shard ❌

```
Shard 12
 ├── Bucket 0
 ├── Bucket 1
 ├── Bucket 2
 └── Bucket 99
```

Advantages:

* Smaller tables/indexes
* Faster local queries
* Easier maintenance

Disadvantage:

* All writes still go to **Shard 12**.
* CPU, memory, disk, and network remain bottlenecked on the same machine.

So this helps with **data organization**, not with **load distribution**.

---

### Case 2: Buckets are part of the sharding key ✅

Instead of:

```
Shard = hash(userId)
```

Use:

```
Shard = hash(userId + bucket)
```

Example:

```
User A

Bucket 0 → Shard 2
Bucket 1 → Shard 7
Bucket 2 → Shard 12
Bucket 3 → Shard 15
```

Now the 500 million writes are spread across multiple shards.

---

### Rule of thumb

* **Buckets inside a shard** → improve storage organization and query performance.
* **Buckets in the shard key** → distribute traffic across shards and solve the hot shard problem.

### In production

Large-scale systems (e.g., messaging or social media platforms) typically use **logical buckets** that are incorporated into the partition/sharding key, rather than just creating buckets inside a single shard. That way, both storage and write load are distributed across the cluster.

Q: How many buckets we need to create and how its mapped to shards?
This is a common system design interview question. The key idea is that **buckets are logical**, while **shards are physical**.

## Example

Suppose you have:

* **16 physical shards**
* **1024 logical buckets**

```
Bucket 0    → Shard 0
Bucket 1    → Shard 1
Bucket 2    → Shard 2
...
Bucket 15   → Shard 15
Bucket 16   → Shard 0
Bucket 17   → Shard 1
...
Bucket 1023 → Shard 15
```

Each shard owns **1024 / 16 = 64 buckets**.

---

## How many buckets?

There is no fixed number. A common practice is to create **far more buckets than shards**.

| Physical Shards | Logical Buckets |
| --------------: | --------------: |
|               8 |        256–1024 |
|              16 |       1024–4096 |
|              32 |       4096–8192 |
|             100 |            16K+ |

The reason is future scalability. If you start with only 16 buckets on 16 shards, adding more shards later requires moving large amounts of data.

---

## How is a bucket chosen?

For a hot user:

```text
bucket = hash(messageId) % 1024
```

or

```text
bucket = hash(campaignId) % 1024
```

Then:

```text
shard = bucket % 16
```

Example:

```
messageId = 987654

bucket = hash(messageId) % 1024 = 517

shard = 517 % 16 = 5
```

So that message is stored on **Shard 5**.

---

## Why not one bucket per shard?

Imagine today:

* 16 shards
* 16 buckets

Tomorrow you add 16 more shards.

You would need to move about **50% of your data**.

With 1024 buckets, you simply reassign some buckets to the new shards:

```
Before

Shard 0
Buckets 0–63

After adding shards

Shard 0
Buckets 0–31

Shard 16
Buckets 32–63
```

Only those buckets are moved, making rebalancing much easier.

---

## Production systems

Distributed databases such as Cassandra, DynamoDB, ScyllaDB, and CockroachDB use a similar concept: many **logical partitions/tokens** are mapped onto a smaller number of **physical nodes**. This allows efficient rebalancing as the cluster grows.

### Interview answer

If asked, "How many buckets would you create for 16 shards?", a solid response is:

> "I would create significantly more logical buckets than physical shards—typically 1024 or 4096 for a 16-shard cluster. Each bucket is assigned to a shard using a mapping layer. When new shards are added, I only remap buckets instead of rehashing the entire dataset, which minimizes data movement and avoids hot spots."

Q: While reading data?
For **reads**, the mapping layer is just as important as it is for writes.

Suppose:

* 16 physical shards
* 1024 logical buckets
* Shard key = `(userId, bucket)`

### If you only know `userId`

You **cannot determine the bucket directly** unless you have additional information. There are three common approaches.

### Option 1: Fan-out read (simple)

```text
for bucket = 0 to 1023
    shard = bucketToShardMap[bucket]
    query(userId, bucket)
merge results
```

This works but can be expensive if there are many buckets.

---

### Option 2: Metadata index (preferred)

Maintain a small metadata table:

```text
User123
Date: 2026-07-14

Buckets used:
5
18
201
512
```

When reading:

```text
Read metadata
      ↓
Buckets = [5,18,201,512]
      ↓
Map each bucket to its shard
      ↓
Read those shards in parallel
      ↓
Merge and return
```

This is the typical production approach because it avoids querying empty buckets.

---

### Option 3: Deterministic bucket

If the bucket is derived from something the client also knows (for example, `campaignId`):

```text
bucket = hash(campaignId) % 1024
```

Then you can directly compute the bucket and shard without scanning or metadata.

---

### For your messaging use case (500M messages)

A good production design would be:

* **Write:** `bucket = hash(messageId or campaignId) % 1024`
* **Store metadata:** `(userId, date) → list of buckets`
* **Read:** Look up the metadata, identify the relevant buckets, query those shards in parallel, and merge the results.

This provides scalable writes while keeping reads efficient. It avoids both hot shards and unnecessary fan-out across every bucket.

Q:Kafka,we have offset id,can I retrieve directly with message from kafka?

Yes. **If you know the exact topic, partition, and offset, you can retrieve that specific message directly from Kafka.**

An offset is **unique only within a partition**, so an offset alone is not enough.

You need:

* **Topic**
* **Partition**
* **Offset**

Then the consumer can seek directly to that offset:

```java
consumer.assign(List.of(new TopicPartition("payments", 2)));
consumer.seek(new TopicPartition("payments", 2), 12345L);

ConsumerRecords<String, String> records = consumer.poll(Duration.ofSeconds(1));
```

If a message exists at offset `12345` in partition `2`, you'll receive it.

### Interview follow-up question

If the interviewer asks:

> "Can you fetch a message using only the offset?"

The correct answer is:

> **No. An offset is meaningful only within a specific partition of a topic. You must know the topic and partition along with the offset.**

### Real-world use cases

* Debugging a failed payment event.
* Reprocessing a specific event.
* Auditing or investigating production issues.
* Recovering from consumer failures.

### One caveat

Kafka retains messages only according to its **retention policy** (time- or size-based). If the message has been deleted due to retention or log compaction, you **cannot retrieve it**, even if you know the topic, partition, and offset.

This is a common Kafka interview question, especially for Platform and Payments engineering roles.

Q:While publishing the message at that time do we get the metadata?

Yes. **When a producer successfully publishes a message to Kafka, the broker returns metadata** in the acknowledgment.

The producer receives a `RecordMetadata` object containing:

* **Topic**
* **Partition**
* **Offset** (assigned by the broker)
* **Timestamp**
* **Leader Epoch** (in newer Kafka versions)
* **Serialized key/value sizes**

Example:

```java
ProducerRecord<String, String> record =
    new ProducerRecord<>("payments", "key1", "Payment Completed");

RecordMetadata metadata =
    producer.send(record).get();

System.out.println(metadata.topic());      // payments
System.out.println(metadata.partition());  // 2
System.out.println(metadata.offset());     // 12345
System.out.println(metadata.timestamp());  // 1753098765432
```

### How it works internally

1. Producer sends the message.
2. Kafka broker appends it to the partition log.
3. Broker assigns the **next available offset**.
4. Broker sends an acknowledgment containing the metadata.
5. The producer receives the `RecordMetadata`.

So **the producer does not know the offset before publishing**. The offset is generated by the broker **after** the message is successfully written.

### Interview follow-up

If asked:

> "Can the producer specify the offset?"

The correct answer is:

> **No. The producer can choose the topic, key, and optionally the partition, but the Kafka broker is solely responsible for assigning the offset.**

This distinction is frequently tested in Kafka interviews.

Q: How to retrieve it during resend for failed message processing ?
This is an excellent interview question. In production systems, **you normally do not retrieve the original message from Kafka using its offset for retries**. Instead, there are well-established retry patterns.

## Pattern 1: Retry from the original message (most common)

When consuming a message:

```
Topic: payments
Partition: 2
Offset: 12345
```

If processing fails:

* Do **not commit the offset**.
* Retry processing locally a few times (e.g., 3 attempts).
* If it still fails, publish the **same message** to a retry topic or DLQ.

The original message is already available in memory while processing, so there's no need to fetch it again by offset.

---

## Pattern 2: Store metadata for audit/debugging

Many systems persist:

```
Topic      : payments
Partition  : 2
Offset     : 12345
Key        : payment123
Headers    : correlationId
Payload    : {...}
```

If you later need to inspect or replay the event, you know exactly where it came from.

---

## Pattern 3: Retry Topics (recommended)

```
payments
    ↓
Processing failed
    ↓
payments.retry.1
    ↓
Still failed
    ↓
payments.retry.2
    ↓
Still failed
    ↓
payments.dlq
```

The retry message usually includes headers such as:

```
originalTopic
originalPartition
originalOffset
retryCount
errorReason
```

This is the pattern used in many enterprise Kafka deployments.

---

## Pattern 4: Fetch by offset (rare)

If you only stored:

```
Topic = payments
Partition = 2
Offset = 12345
```

you can create a consumer, call `seek(partition, offset)`, and `poll()` to retrieve that message—**provided it hasn't been deleted by Kafka's retention policy**.

This approach is mainly used for:

* Production debugging
* Event replay
* Auditing

It is **not** the standard mechanism for implementing retries.

### Interview takeaway

If asked:

> "How do you resend a failed Kafka message?"

A strong answer is:

> "During processing, I already have the payload. I retry locally first. If retries are exhausted, I publish the same payload to a retry topic or DLQ while preserving metadata such as the original topic, partition, offset, and correlation ID in the message headers. I don't normally fetch the original message again by offset because it adds unnecessary complexity and may fail if the message has been removed due to Kafka retention."

That's the answer most interviewers expect for a production-grade Kafka system.
