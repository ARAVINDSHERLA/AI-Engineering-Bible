---

# 1. Why use RAG instead of fine-tuning?

**Use RAG when:**

* Knowledge changes frequently.
* Need enterprise/private documents.
* Want citations and traceability.
* Lower cost than retraining.

**Use Fine-tuning when:**

* Want new behavior or writing style.
* Domain-specific reasoning.
* Task specialization.

**Best practice:** Combine both—fine-tune for behavior, RAG for knowledge.

---

# 2. How do you decide chunk size and overlap?

Depends on:

* Document type
* Embedding model context window
* LLM context size

Typical values:

* Chunk size: **300–800 tokens**
* Overlap: **10–20% (50–150 tokens)**

Small chunks → better retrieval but lose context.

Large chunks → more context but lower retrieval precision.

Semantic chunking usually outperforms fixed-size chunking.

---

# 3. What causes hallucinations in RAG?

Common causes:

* Poor retrieval
* Wrong chunks
* Missing documents
* Irrelevant context
* Weak prompts
* Model guessing

Reduce by:

* Better chunking
* Hybrid search
* Reranking
* Metadata filtering
* Prompt grounding
* Confidence thresholds
* Answer only from retrieved context

---

# 4. When should you use vector search, keyword search, or hybrid search?

**Vector Search**

* Semantic meaning
* Natural language queries

**Keyword Search (BM25)**

* IDs
* Error codes
* Product names
* Exact terms

**Hybrid Search**

* Production systems
* Combines semantic + lexical
* Usually gives the best results

---

# 5. Why is reranking important?

Vector search retrieves approximately relevant documents.

A reranker:

* Reads the query and document together.
* Produces a better relevance score.
* Reorders top results.

Benefits:

* Higher Precision@K
* Fewer hallucinations
* Better answer quality

---

# 6. How do embeddings work? What makes a good embedding model?

Embeddings convert text into dense vectors where semantically similar texts are close in vector space.

A good embedding model:

* Captures semantic meaning
* Supports multilingual text (if needed)
* High retrieval accuracy
* Fast inference
* Small memory footprint

Examples:

* BAAI BGE
* E5
* OpenAI embeddings
* Voyage AI
* Jina AI

---

# 7. How do you evaluate retrieval quality?

**Precision@K**

* Relevant documents in top K.

**Recall@K**

* How many relevant documents were retrieved.

**MRR (Mean Reciprocal Rank)**

* Measures how early the first correct document appears.

**nDCG**

* Rewards highly relevant documents appearing higher in the ranking.

Higher values indicate better retrieval.

---

# 8. How do you evaluate answer quality and faithfulness?

Metrics:

* Correctness
* Relevance
* Completeness
* Faithfulness (grounded in retrieved context)
* Hallucination rate
* Citation accuracy

Tools:

* RAGAS
* DeepEval
* LangSmith
* Phoenix
* Human evaluation

---

# 9. How do you optimize latency in a production RAG pipeline?

Optimize each stage:

* Faster embeddings
* Better vector indexing (HNSW/IVF)
* Cache embeddings and responses
* Reduce retrieved documents
* Efficient reranker
* Smaller or distilled LLM
* Parallel retrieval
* Streaming responses
* GPU inference
* Quantization

Measure latency at every stage.

---

# 10. When should you improve retrieval instead of moving to a larger LLM?

Improve retrieval first if:

* Wrong documents are retrieved.
* Context is incomplete.
* Hallucinations are caused by missing knowledge.
* Domain-specific data isn't found.

Move to a larger LLM only if:

* Reasoning is insufficient despite good retrieval.
* Complex multi-step reasoning is required.

Better retrieval often gives larger gains at lower cost than a bigger model.

---

# 11. How do you handle continuously changing knowledge without retraining?

Use:

* Incremental indexing
* Document versioning
* Event-driven ingestion
* CDC (Change Data Capture)
* Scheduled re-indexing
* Metadata filtering
* Cache invalidation
* Knowledge freshness checks

RAG enables updating knowledge by refreshing the index rather than retraining the model.

# Dense vs Sparse Embeddings

| Feature           | Dense Embeddings                                | Sparse Embeddings                                 |
| ----------------- | ----------------------------------------------- | ------------------------------------------------- |
| Representation    | Dense floating-point vector                     | Mostly zeros, few non-zero values                 |
| Captures          | Semantic meaning                                | Exact keywords and lexical matches                |
| Dimension         | 384–3072 values                                 | Tens of thousands of dimensions (vocabulary size) |
| Generated by      | Neural networks (BERT, E5, BGE, OpenAI, Voyage) | TF-IDF, BM25, SPLADE                              |
| Similarity        | Cosine similarity, dot product                  | Keyword overlap/scoring                           |
| Storage           | Compact                                         | Larger but sparse representation                  |
| Handles synonyms  | ✅ Excellent                                     | ❌ Poor                                            |
| Handles exact IDs | ❌ Weak                                          | ✅ Excellent                                       |
| Best for          | Semantic search                                 | Keyword search                                    |

---

## Dense Embeddings

A dense embedding converts text into a vector where **every dimension has a meaningful value**.

Example:

```
"What is Kafka?"

↓

[0.12, -0.84, 0.67, 0.91, ...]
```

A sentence with similar meaning:

```
"Explain Apache Kafka"

↓

[0.15, -0.82, 0.69, 0.88, ...]
```

Although the words differ, the vectors are close, so semantic search retrieves the right document.

### Advantages

* Understands meaning
* Handles synonyms
* Supports multilingual search
* Great for natural language queries

### Disadvantages

* May miss exact identifiers like `ERR_503`
* Can confuse similar concepts if context is limited
* Requires an embedding model

---

## Sparse Embeddings

Sparse embeddings assign weights only to important terms; most dimensions are zero.

Example vocabulary:

```
["Kafka", "Redis", "Java", "Spring", "Python"]
```

Document:

```
"Kafka with Java"
```

Vector:

```
[0.95, 0, 0.83, 0, 0]
```

# Interview Answer (60 seconds)

> "Dense embeddings are continuous vectors generated by neural networks that capture semantic similarity, making them ideal for natural language search and paraphrases.
> Sparse embeddings represent text using weighted keywords, making them excellent for exact term matching such as product IDs, error codes, and technical keywords.
> Dense retrieval excels at semantic understanding, while sparse retrieval excels at lexical matching. In production RAG systems, we typically use **hybrid search**,
>  combining both dense and sparse retrieval—often with Reciprocal Rank Fusion (RRF)—followed by a reranker.
> This approach consistently improves retrieval accuracy and reduces hallucinations compared to using either method alone."

This is the level of explanation expected in senior AI Engineer and LLM Engineer interviews.

### So what is the LLM responsible for?

The LLM is **not** responsible for:

* Finding the correct document
* Applying user permissions
* Filtering departments or tenants
* Determining whether the user is authorized
* Retrieving relevant chunks

Those are deterministic engineering tasks.

The LLM **is** responsible for:

* Synthesizing information from retrieved documents
* Answering natural language questions
* Explaining concepts
* Summarizing multiple sources
* Reasoning across retrieved evidence
* Producing structured outputs if needed

### Example

Suppose the user asks:

> "What is our leave policy for maternity leave?"

Before the LLM is called, the system should already have:

* Verified the user's identity.
* Confirmed they are allowed to access HR policies.
* Retrieved only the latest HR policy document.
* Selected the relevant sections on maternity leave.

The LLM then receives only those relevant sections and answers the question. It is **grounded** in the retrieved context rather than searching the enterprise knowledge itself.

### Can we skip the LLM?

Yes, for some use cases.

If the query is:

* "Show policy HR-102."
* "What is employee ID 12345?"
* "Return invoice INV-2025-001."

A deterministic retrieval system can return the result directly without an LLM.

Use the LLM when the user asks:

* "Summarize the leave policy."
* "Compare the travel policy and work-from-home policy."
* "What are the key changes between version 2 and version 3?"
* "Explain this technical design document in simple terms."

### Interview perspective

A strong answer in an AI Engineer interview is:

> "In production Enterprise RAG, we minimize the LLM's responsibility. Authentication, authorization, query validation, metadata filtering, retrieval, and reranking are deterministic components executed before invoking the LLM. The LLM's role is limited to reasoning, summarization, and answer generation based on trusted, retrieved context. This architecture reduces hallucinations, improves latency, lowers cost, and provides more predictable behavior."

That distinction—**using deterministic systems for retrieval and the LLM for reasoning**—is a hallmark of well-designed enterprise AI systems.

## 3. How do HNSW, IVF, and PQ indexing work?

* **HNSW:** Graph-based index for very fast Approximate Nearest Neighbor (ANN) search; high accuracy, more memory.
* **IVF (Inverted File):** Clusters vectors first, then searches only relevant clusters; faster on large datasets.
* **PQ (Product Quantization):** Compresses vectors to reduce memory with a small accuracy trade-off.

Q: When we have tool calling,why mcp is required?

This is one of the most important concepts in AI Engineering interviews. Many people think **MCP replaces tool calling**, but that's not true.

**Tool Calling answers: *"How does an LLM invoke a tool?"***

**MCP answers: *"How do we standardize and manage thousands of tools, resources, and prompts across applications?"***

Think of it this way:

### Without MCP

The LLM knows about every tool directly.

```
LLM
 ├── search_orders()
 ├── get_customer()
 ├── create_ticket()
 ├── send_email()
 ├── query_database()
 ├── github_search()
 ├── jira_search()
 ├── slack_send()
 └── ...
```

Each tool requires:

* Custom JSON schema
* Authentication
* Error handling
* Version management
* Discovery
* Client implementation

If another team builds a new tool, every client must be updated.

---

### With MCP

The LLM talks to a single protocol.

```
LLM
   │
MCP Client
   │
MCP Server
   ├── GitHub
   ├── Jira
   ├── Database
   ├── SAP
   ├── Salesforce
   ├── File System
   └── Internal APIs
```

The model doesn't need to know implementation details. It asks the MCP server:

> "What tools do you have?"

The server responds with tool definitions and capabilities.

# Interview Answer (2-minute version)

> Tool calling enables an LLM to invoke a specific function by passing structured arguments. It solves the problem of function execution. MCP operates at a higher abstraction level. It standardizes how AI applications discover, authenticate with, and interact with tools, resources, prompts, and sessions across different systems. In an enterprise with hundreds of internal services and many AI agents, MCP avoids building custom integrations for every application by providing a common protocol. In other words, tool calling is a feature, while MCP is the ecosystem and protocol that makes tool usage reusable, discoverable, and scalable.

For the kind of enterprise AI platforms you're preparing to build, think of the layers as:

```
LLM
   ↓
Reasoning & Planning
   ↓
Tool Calling (decides which function to invoke)
   ↓
MCP Client (speaks the MCP protocol)
   ↓
MCP Server (discovers tools, enforces auth, manages resources)
   ↓
Enterprise Systems (SAP, Jira, GitHub, Databases, etc.)
```

So tool calling and MCP are complementary: **tool calling decides *what* to execute, while MCP standardizes *how* enterprise capabilities are exposed and accessed.**

Q:How agent ai or ai systems understands knowledge graph, actually what will have kg?

This is one of the most important concepts in enterprise AI. Many people say "use a Knowledge Graph," but very few explain **what actually goes into it** and **how an AI agent uses it**.

The key idea is:

> A Knowledge Graph (KG) does not store documents. It stores **entities, relationships, properties, and business semantics**.

Think of it as the enterprise's **digital map**, while a vector database is the enterprise's **semantic memory**.

## Example: Dell Order Management

Suppose a user asks:

The raw documents may contain:

* SAP order
* Shipment details
* Warehouse inventory
* Supplier email
* Jira ticket
* Slack discussion
* Invoice

A Knowledge Graph extracts structured facts like:

   │

   │

   │

   │

   │

   │

Each box is a **node (entity)**, and each arrow is a **relationship**.

## Why combine KG with a Vector DB?

Each solves a different problem:

| Knowledge Graph                        | Vector Database                       |
| -------------------------------------- | ------------------------------------- |
| Stores entities and relationships      | Stores semantic embeddings of text    |
| Answers "How are things connected?"    | Answers "Which passages are similar?" |
| Excellent for reasoning and traversals | Excellent for semantic retrieval      |
| Highly structured                      | Unstructured or semi-structured       |
| Uses graph traversal                   | Uses nearest-neighbor search          |

The strongest enterprise AI systems use **both**:

1. Use the **Knowledge Graph** to discover the relevant business entities and their relationships.
2. Use the **Vector Database** to retrieve the supporting documents, emails, specifications, or tickets.
3. Feed both structured graph facts and retrieved evidence into the LLM.

This combination—often called **Graph RAG**—is increasingly used in complex enterprise domains such as supply chain, manufacturing, finance, healthcare,
and enterprise knowledge systems because it supports both reasoning over relationships and grounding answers in source documents.


Q: Where facts,rules where saved and understands the ai systems ?

This is one of the biggest misconceptions in GenAI.

**The LLM itself does NOT store your enterprise facts or business rules.** Instead, a production AI system assembles knowledge from multiple specialized stores before calling the LLM.

Think of it as a "knowledge fabric" rather than a single database.

## Where are facts stored?

Facts are objective pieces of information.

Examples:

* Customer ABC placed Order 12345.
* Dell acquired EMC.
* Order status = Shipped.
* Employee John belongs to Finance.

These are typically stored in:

* Transactional databases (PostgreSQL, Oracle, SQL Server, MySQL)
* Data warehouses/lakehouses (Snowflake, Databricks, Microsoft Fabric)
* Knowledge Graphs (Neo4j, Amazon Neptune)
* Vector databases (linked to supporting documents, not as the primary fact store)

Example:

```text
Order 12345
status = Delayed
customer = ABC Corp
warehouse = Bangalore
```

---

## Where are business rules stored?

Rules define how the business operates.

Examples:

* Orders above $100,000 require VP approval.
* Gold trades above ₹50 lakh need compliance review.
* Recyclable plastic must be processed within 30 days.

These usually live in:

* Rule engines (Drools, IBM ODM)
* Policy engines (Open Policy Agent, Cedar)
* Decision tables
* Configuration services
* Workflow engines (Camunda, Temporal)
* Sometimes in a Knowledge Graph if they are relationship-based

Example:

```text
IF amount > 100000
THEN approval = VP
```

---

This is what RAG retrieves.

Examples:

* PDFs
* Word documents
* Confluence pages
* SharePoint
* Slack messages
* GitHub READMEs

↓

### Architect interview perspective

The important insight is that **Multimodal RAG is not fundamentally different from text RAG**. The difference lies in the ingestion and understanding stages:

* **Text** → chunk and embed.
* **Audio** → transcribe, segment by speaker/topic, then embed.
* **Video** → extract frames, transcribe audio, detect scenes, caption visuals, then embed.
* **Images** → OCR + visual understanding + embeddings.
* **CAD** → parse components and relationships, then build a knowledge graph.

After normalization, all these modalities contribute structured context that the retrieval and reasoning pipeline can use together. This unified approach is what modern enterprise AI platforms use to answer questions across documents, meetings, videos, images, and operational data.

## Interview question: "Do we store only embeddings?"

The best answer is:

> **No.** A production vector database stores the embedding along with the original content (or a reference to it), metadata, and a unique identifier.
> The embedding enables semantic similarity search, while metadata supports filtering (for example, by document, page, department, or access level).
> The original content is what is ultimately retrieved and provided to the LLM as context. Large binary assets such as PDFs and images are typically
> stored in object storage (S3/MinIO), with references maintained in the vector database.

### Interview talking points

* Predictions must be asynchronous and complete in milliseconds.
* Features should come from an online feature store (e.g., Redis or Feast) to ensure consistency between training and inference.
* Store every prediction with the model version and feature values for auditability.
* Retrain only when sufficient new labeled data arrives or when drift/performance thresholds are exceeded.
* Validate the new model against the current production model before deployment.
* Roll out using shadow, canary, or blue-green deployment strategies.
* Continuously monitor latency, throughput, precision, recall, false-positive rate, false-negative rate, and concept drift in production.

Q: means this is purely reactive+adapatative usecase ?
Yes. The architecture above is **primarily a reactive, adaptive AI/ML system**, not a proactive or agentic one.

Here's how to classify it:

| Capability             | Present? | Explanation                                                                                                      |
| ---------------------- | -------- | ---------------------------------------------------------------------------------------------------------------- |
| **Reactive**           | ✅ Yes    | Scores each transaction as it arrives and immediately returns a decision (approve/reject/review).                |
| **Adaptive**           | ✅ Yes    | Learns from newly labeled data (fraud/not fraud) and retrains when drift or performance degradation is detected. |
| **Predictive**         | ✅ Yes    | Predicts the probability of fraud before the outcome is known.                                                   |
| **Proactive**          | ❌ No     | Does not initiate actions before a transaction occurs or anticipate future fraud campaigns.                      |
| **Autonomous/Agentic** | ❌ No     | Does not plan, reason, use tools, or make multi-step decisions independently.                                    |

### Reactive loop

```text

        ↓

        ↓
ML Model
        ↓

        ↓
Approve / Reject / Review
```

The system only acts **after receiving an event**, making it reactive.

Q: ensemble, bagging, bootstrapping and stacking?

These are fundamental ensemble learning concepts and are frequently asked in ML interviews. The key is to understand **how models are trained and how their predictions are combined**.

---

# 1. Ensemble Learning

**Definition:**
Ensemble learning combines multiple models (called **base learners**) to produce a better prediction than any individual model.

**Idea**

Instead of relying on one expert, ask multiple experts.

```text
          Model 1
             │
          Model 2
             │
Input →   Model 3  → Combine → Final Prediction
             │
          Model 4
```

### Advantages

* Higher accuracy
* Better generalization
* Reduced overfitting
* More robust predictions

### Types

* Bagging
* Boosting
* Stacking
* Voting

---

# 2. Bagging (Bootstrap Aggregating)

Bagging trains **many models independently and in parallel**.

Each model sees a **different random subset** of the training data.

```text
             Dataset
                │
    --------------------------
    │      │      │      │
 Sample1 Sample2 Sample3 Sample4
    │      │      │      │
 Tree1  Tree2  Tree3  Tree4
     \     |      |     /
       Majority Vote
```

### Characteristics

* Models are independent.
* Training happens in parallel.
* Final prediction uses voting (classification) or averaging (regression).
* Mainly reduces **variance**.

---

## Example

Random Forest is bagging.

100 decision trees

Each tree:

* Different rows
* Different features

Final prediction

```
Tree1 → Fraud

Tree2 → Fraud

Tree3 → Genuine

Tree4 → Fraud

Final = Fraud
```

---

# 3. Bootstrapping

Bootstrapping is **not an ensemble method**.

It is a **sampling technique** used by bagging.

Suppose we have:

```
Dataset

A B C D E
```

Random sampling **with replacement**

```
Sample 1

A B C C E

Sample 2

A D D B C

Sample 3

E B B C D
```

Notice:

* Some records appear multiple times.
* Some records are missing.
* Every sample has the same size as the original dataset.

This is bootstrapping.

---

## Why "with replacement"?

Example:

```
Original

1
2
3
4
5
```

Bootstrap sample

```
2
4
2
5
1
```

Number 2 appears twice.

Number 3 is absent.

---

# Bagging = Bootstrapping + Aggregation

Bagging uses:

```
Bootstrap samples
        ↓
Train multiple models
        ↓
Aggregate predictions
```

Hence:

**Bagging = Bootstrap + Aggregating**

---

# 4. Stacking

Stacking is completely different.

Instead of voting,

one model **learns how to combine other models**.

```text
              Dataset
                  │
      ------------------------
      │          │          │
     XGBoost   RandomForest  SVM
      │          │          │
      └──── Predictions ────┘
                │
          Meta Model
       (Logistic Regression)
                │
          Final Prediction
```

The top model is called:

* Meta learner
* Level-2 model

The bottom models:

* Base learners
* Level-1 models

---

## Example

Transaction

```
XGBoost predicts

0.90

Random Forest predicts

0.70

Neural Network predicts

0.85
```

Meta model learns

```
When XGBoost >0.8
and RF >0.6

Fraud probability is 95%
```

# How stacking works

Step 1

Train

```
Random Forest

XGBoost

SVM
```

Step 2

Generate predictions

```
RF → 0.72

XGB → 0.88

SVM → 0.79
```

Step 3

Use predictions as features

```
0.72
0.88
0.79
```

Train another model.

That model makes the final prediction.

# Bagging vs Stacking

| Bagging                               | Stacking                                                   |
| ------------------------------------- | ---------------------------------------------------------- |
| Same algorithm (often decision trees) | Different algorithms                                       |
| Parallel training                     | Base models train in parallel; meta-model trains afterward |
| Voting/Average                        | Meta learner combines predictions                          |
| Reduces variance                      | Learns optimal combination                                 |
| Example: Random Forest                | Example: XGBoost + RF + SVM + Logistic Regression          |

---

# Bagging vs Boosting

| Bagging            | Boosting                                                 |
| ------------------ | -------------------------------------------------------- |
| Parallel           | Sequential                                               |
| Independent models | Each model corrects previous errors                      |
| Reduces variance   | Reduces bias (and can reduce variance)                   |
| Equal importance   | Later models focus on difficult samples                  |
| Random Forest      | AdaBoost, Gradient Boosting, XGBoost, LightGBM, CatBoost |

---

# Complete Relationship

```text
                    Ensemble Learning
                           │
      ┌────────────────────┼───────────────────┐
      │                    │                   │
   Bagging             Boosting           Stacking
      │
      │
Bootstrapping
(Sampling with replacement)
```

## Interview one-liners

* **Ensemble Learning:** Combines multiple models to improve predictive performance and robustness.
* **Bootstrapping:** Creates multiple training datasets by sampling with replacement from the original data.
* **Bagging:** Trains multiple independent models on bootstrap samples and combines their predictions using voting or averaging.
* **Stacking:** Trains multiple base models and uses a meta-model to learn the best way to combine their predictions.
* **Boosting:** Trains models sequentially, where each new model focuses on correcting the mistakes of the previous ones.

### Fraud detection example

Suppose you're building a real-time fraud detection system:

* **Bagging:** Use a Random Forest with hundreds of decision trees for a fast, robust fraud score.
* **Boosting:** Use XGBoost or LightGBM to achieve higher predictive accuracy on historical fraud data.
* **Stacking:** Combine XGBoost, Random Forest, and a Neural Network, then train a logistic regression meta-model on their outputs to produce the final fraud probability.

This progression—from bagging to boosting to stacking—reflects increasing sophistication in how multiple models are combined.

Q:Data at rest and data at motion secure how?

This is a very common security interview question, especially for AI/ML, cloud, data engineering, and distributed systems.

## 1. Data at Rest

**Definition:** Data stored on persistent storage.

Examples:

* Database (MySQL, PostgreSQL, MongoDB)
* Data Lake (S3, ADLS, MinIO, HDFS)
* Data Warehouse
* Vector Database
* Files (PDF, Parquet, Iceberg, Delta)
* Backups and snapshots

### How to secure data at rest

| Technique                 | Description                                                                       |
| ------------------------- | --------------------------------------------------------------------------------- |
| Encryption                | Encrypt data before storing using AES-256 (industry standard).                    |
| Disk Encryption           | Encrypt EBS volumes, Azure Managed Disks, SSDs, HDFS disks.                       |
| Database Encryption       | Transparent Data Encryption (TDE) for relational databases.                       |
| Object Storage Encryption | Server-side or client-side encryption for S3, ADLS, MinIO.                        |
| Key Management            | Store encryption keys in KMS, HSM, or Azure Key Vault; never in application code. |
| Access Control            | Use IAM, RBAC, ACLs, least privilege.                                             |
| Data Masking              | Mask or tokenize sensitive fields such as Aadhaar or credit card numbers.         |
| Auditing                  | Log all access to sensitive data.                                                 |

### Example

```text
Application
      │
AES-256 Encrypt
      │
S3 / MinIO / ADLS
```

Even if someone steals the storage, they cannot read the data without the encryption key.

---

# 2. Data in Motion (Data in Transit)

**Definition:** Data moving between systems over a network.

Examples:

* Browser → API
* API → Database
* Microservice → Microservice
* Kafka Producer → Broker
* Broker → Consumer
* Spark → HDFS
* LLM → Vector Database

### How to secure data in transit

| Technique              | Description                                                   |
| ---------------------- | ------------------------------------------------------------- |
| TLS/SSL                | Encrypt network traffic using HTTPS or TLS.                   |
| mTLS                   | Mutual TLS so both client and server authenticate each other. |
| VPN                    | Secure communication between networks or data centers.        |
| IPSec                  | Encrypt network-layer traffic between hosts or sites.         |
| SSH                    | Secure administrative access and file transfers.              |
| Certificate Management | Use trusted X.509 certificates and rotate them regularly.     |
| API Authentication     | OAuth 2.0, JWT, API keys, OpenID Connect.                     |
| Message Encryption     | Encrypt Kafka, RabbitMQ, or gRPC traffic with TLS.            |

### Example

```text
Client
   │
HTTPS (TLS 1.3)
   │
Spring Boot API
   │
TLS
   │
Database
```

Anyone intercepting the traffic sees only encrypted data.

# Interview one-liner

> **Data at rest** is protected by encrypting stored data (typically with AES-256), enforcing least-privilege access through IAM/RBAC, managing keys securely with KMS/HSM, and auditing access. **Data in motion** is protected by encrypting network traffic using TLS/mTLS, authenticating communicating parties, and securing APIs and messaging systems with protocols such as HTTPS, gRPC over TLS, and Kafka with TLS.





This distinction is important: **the vector database is a semantic index**, not the primary storage for all enterprise content. 
It helps locate relevant information; the LLM reasons over the retrieved content, not over the embedding vectors themselves.


This is the workflow typically expected for production-grade document intelligence systems rather than simple fixed-size text chunking.
